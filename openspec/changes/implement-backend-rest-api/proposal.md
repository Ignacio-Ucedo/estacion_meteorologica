## Why

The frontend and future gateway need a concrete REST backend that implements the existing station and weather-reading contract. Implementing the FastAPI API now turns the current backend specification into executable behavior with generated OpenAPI documentation and a persistence layer suitable for historical weather data.

## What Changes

- Add a Python 3.13+ FastAPI backend application under the repository with async endpoints under the `/api` prefix.
- Implement station registration, station listing, station detail with latest reading, paginated reading history, hourly metric series, and daily metric summaries according to `openspec/specs/backend-API/spec.md`.
- Add async SQLAlchemy 2.x models and Alembic migrations for PostgreSQL + TimescaleDB-compatible storage.
- Add Pydantic v2 request and response schemas with camelCase JSON responses and ISO-8601 datetime serialization.
- Add tests for endpoint behavior, validation errors, pagination, metric filtering, and aggregation response shapes.
- Preserve `/docs` and `/openapi.json` generated by FastAPI.

## Impact

- Affected component: backend.
- New backend code, dependency metadata, database models, migrations, and test suite.
- API surface exposed under `/api`, consumed by the React frontend and later by the LoRa gateway.
- Runtime dependencies include FastAPI, Pydantic v2, SQLAlchemy 2.x async support, Alembic, PostgreSQL async driver, and Pandas.
- No firmware payload change, LoRa message frequency change, or field-deployed firmware rollback plan is required for this backend-only change.
